# Greasy Fork 手动上传与自动化规划

## 结论

推荐路线是：

```text
先手动创建 Greasy Fork 脚本页
  -> 在 Greasy Fork 配置同步来源
  -> 在 Greasy Fork 生成 Webhook Secret
  -> 在 GitHub 配置原生 Release Webhook 和相同的 Secret
  -> 后续用 build publish 自动发布 Release + Pages，由 GitHub 通知 Greasy Fork
```

第一次通常需要人工创建 Greasy Fork 脚本页，因为要确认脚本说明、分类、license、隐私说明和 LLM 请求说明。

当前 Greasy Fork 页面：

```text
https://greasyfork.org/zh-CN/scripts/586466-tampermonkey-plugin-koishi-market-ai-helper
```

后续可以让 Greasy Fork 从 GitHub Release latest 固定 `.user.js` URL 同步新版本。

## 官方信息摘记

```text
https://greasyfork.org/en/help/api
https://greasyfork.org/en/help/meta-keys
https://greasyfork.org/en/users/sign_in
```

当前判断：

1. Greasy Fork JSON API 是只读 API，不适合做 CI 直接发布。
2. Greasy Fork 提供 prefill 表单，但需要用户 session cookie，不能当成正式 CI 发布 API。
3. Greasy Fork 支持基于 GitHub / GitLab / Bitbucket 的 webhook / sync 检查更新。
4. 从 Greasy Fork 安装的脚本会由 Greasy Fork 自己负责更新。
5. Greasy Fork 会处理 `@updateURL`、`@installURL`、`@downloadURL`，不要依赖这些字段控制 Greasy Fork 安装源的更新。

## 一次性准备

仓库当前已补齐：

1. `LICENSE`，采用 MIT License。
2. README 隐私说明。
3. README LLM 请求说明。
4. README API key 存储说明。
5. README 跨域请求域名说明。
6. userscript metadata 的 `@license`。
7. userscript metadata 的 `@homepageURL`。
8. userscript metadata 的 `@supportURL`。

手动上传前先跑：

```bash
npm run check
```

确认产物存在：

```text
dist/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

## 手动上传第一版

### 1. 登录 Greasy Fork

打开并登录：

```text
https://greasyfork.org/
```

### 2. 创建新脚本

进入新脚本页面，上传或粘贴：

```text
dist/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

如果页面提供从 URL 导入，也可以尝试填 GitHub Pages 固定 URL：

```text
https://vincentzyuapps.github.io/tampermonkey-plugin-koishi-market-ai-helper/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

如果 Pages 还没启用，先用本地构建产物粘贴上传。

脚本类型选择：正式发布选“公开用户脚本”；测试流程可先选 “Unlisted user script”；不选“库”，因为本项目不是给其他脚本通过 `@require` 调用的依赖库。

### 3. 填写脚本信息

建议信息：

```text
名称：Koishi Market AI Helper
中文名称：Koishi 插件 AI 搜索助手
描述：Koishi 插件市场的 AI 对话式搜索助手，支持本地启发式召回和 LLM 推荐。
分类：工具 / 开发工具 / 生产力，具体以 Greasy Fork 当前选项为准。
许可证：MIT。
```

说明里必须写清：

1. 脚本会读取 Koishi 插件市场 registry 元数据。
2. 用户输入的搜索需求可能发送到用户配置的 LLM API。
3. API key 可选择保存在 Tampermonkey 本地存储。
4. 聊天历史默认不保存，用户可在设置中开启。
5. 脚本不会主动上传 Koishi 登录凭据。

### 4. 提交并记录信息

提交后记录：

```text
Greasy Fork script id:
Greasy Fork script URL:
Greasy Fork sync 页面路径:
Greasy Fork webhook URL:
```

其中 webhook URL 如果页面暂时没给出，就先记录为空，后面在同步设置页面继续找。

### 5. 安装测试

从 Greasy Fork 页面安装一次，确认：

1. Tampermonkey 能识别脚本。
2. `@version` 是当前 `package.json` 版本。
3. 在 Koishi 官方市场页面能显示浮窗。
4. 在 `/market` 路径的 Koishi 控制台页面能兼容运行。

## Greasy Fork 同步设置

### 管理页字段建议

进入：

```text
https://greasyfork.org/zh-CN/scripts/586466-tampermonkey-plugin-koishi-market-ai-helper/admin
```

建议设置：

| 字段 | 建议 |
| --- | --- |
| 推荐脚本 | 先留空 |
| 此脚本的同步方式 | 自动 |
| 同步安装链接 | 优先填 GitHub Pages `.user.js` URL |
| 默认附加信息 | 选 Markdown |
| 作者 / 共同作者 | 先只保留当前账号 |
| 默认语言 | 中文页面用中文说明即可 |

附加信息建议手动维护一版稳定短说明，不建议用 CI 自动同步 GitHub README。

建议 Markdown：

```markdown
Koishi 插件市场的 AI 对话式搜索助手，支持本地启发式召回、OpenAI-compatible 与 Anthropic。

完整说明、隐私策略、LLM 请求说明和更新日志请查看 GitHub README：

https://github.com/VincentZyuApps/tampermonkey-plugin-koishi-market-ai-helper#readme
```

### 推荐同步来源

优先使用 GitHub Pages 固定 URL：

```text
https://vincentzyuapps.github.io/tampermonkey-plugin-koishi-market-ai-helper/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

原因：URL 固定，指向构建后的单文件 userscript，不需要提交 `dist/`，且和 `build publish` 的产物一致。

备用来源：

```text
https://github.com/VincentZyuApps/tampermonkey-plugin-koishi-market-ai-helper/releases/latest/download/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

latest release URL 有重定向，是否能被 Greasy Fork 稳定同步要实际测试。

### 管理页检查建议

Greasy Fork 当前可能提示：

1. 缺少 `@license`：下一次用新构建产物同步后应消失，因为仓库已补 MIT metadata。
2. `@include` 使用正则：当前匹配 `http://任意主机/market`，用于兼容本地和局域网 Koishi 控制台；官方 `https://koishi.chat/zh-CN/market/` 使用更明确的 `@match`。
3. 缺少附加信息：在默认附加信息里填上方 Markdown。

### 同步触发

在 Greasy Fork 的 `/users/webhook-info` 页面生成 Webhook Secret，并把页面提供的 Payload URL 和 Secret 填入 GitHub 原生 Repository Webhook。创建新 Release 时，GitHub 会发送带 HMAC 签名的 `release: published` 事件；Actions 不再重复主动 POST。

## GitHub CI 自动化

### 当前 CI 行为

`build publish` 会执行：

1. 检查提交信息。
2. `npm ci`。
3. `npm run check`。
4. 创建或更新 GitHub Release。
5. 上传 Release assets。
6. 部署 GitHub Pages。

当新版本首次创建 GitHub Release 时，GitHub 原生 Webhook 会通知 Greasy Fork。同版本 Release 更新只产生 `release: edited`，不会发布新的 Greasy Fork 版本。

### 配置 GitHub 原生 Webhook

先登录 Greasy Fork 并打开：

```text
https://greasyfork.org/zh-CN/users/webhook-info
```

点击 `Generate` 或 `Regenerate`，复制页面提供的 Payload URL 和 Webhook Secret。

进入 GitHub 仓库：

```text
Settings -> Webhooks -> Add webhook
```

将 Content type 设为 `application/json`，Secret 填入 Greasy Fork 生成的值，只选择 `Releases` 事件并保持 Active。这个 Secret 不属于 Actions Repository Secret，不要写入仓库文件或日志。

### 自动发布流程

更新版本：

```bash
npm version patch --no-git-tag-version
```

检查并提交：

```bash
npm run check
git add -A
git commit -m "build publish: release and deploy userscript page"
git push origin main
```

预期链路：

```text
GitHub Actions
  -> Release assets 更新
  -> Pages userscript 更新
  -> GitHub 原生 Release Webhook 发送签名事件
  -> Greasy Fork 检查同步来源
  -> Greasy Fork 发现 @version 递增
  -> Greasy Fork 更新脚本版本
```

## 后续手动更新流程

如果自动同步没配置好，可以手动更新：

1. `npm version patch --no-git-tag-version`。
2. `npm run check`。
3. 打开 Greasy Fork 脚本更新页面。
4. 上传或粘贴新的 `dist/*.user.js`。
5. 确认 `@version` 比上一版递增。
6. 提交更新并安装测试。

## 不推荐方案

### GitHub Actions 保存 Greasy Fork cookie

1. Greasy Fork 没有正式写入 API。
2. prefill 需要用户 session cookie。
3. cookie 泄露风险高。
4. CI 只能预填表单，仍不能可靠完成发布。

### Playwright 模拟网页发布

1. 需要保存账号登录态。
2. 页面字段变化会导致脚本脆弱。
3. 失败时很难判断是页面变化、账号风控还是脚本问题。
4. 不如 Greasy Fork 官方 sync / webhook 稳定。

## 验收清单

1. Greasy Fork 页面能展示中文标题和描述。
2. Greasy Fork 页面能展示 license。
3. Greasy Fork 页面说明包含 LLM 请求和 API key 存储策略。
4. 从 Greasy Fork 安装后脚本能在 Koishi 市场页面运行。
5. `build publish` 能完成 Release 和 Pages。
6. GitHub 原生 Webhook 的 `release: published` 投递返回 `2xx`。
7. `@version` 递增后 Greasy Fork 能同步新版本。
