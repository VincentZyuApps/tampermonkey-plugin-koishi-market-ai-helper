# Koishi Market AI Helper 分发规划

## 目标

为 Tampermonkey 脚本提供从低成本到高曝光的分发路径。

本规划按“5 个实际分发渠道 + 2 个发现/入口层”整理。分发渠道负责托管 `.user.js`，发现/入口层负责让用户更容易找到安装链接。

## 总览

### 5 个分发渠道

1. GitHub Release latest download。
2. GitHub Pages 固定 URL。
3. Greasy Fork。
4. OpenUserJS。
5. GitHub / Gist Raw。

### 2 个发现/入口层

1. README 与 Release body 一键安装入口。
2. Userscript.Zone 搜索发现。

## 渠道一：GitHub Release latest download

### 定位

当前第一优先级，负责版本归档、changelog、资产校验和稳定下载。

GitHub 支持 latest release asset 直链：

```text
https://github.com/OWNER/REPO/releases/latest/download/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

参考：

```text
https://docs.github.com/en/repositories/releasing-projects-on-github/linking-to-releases
```

### 要做

1. 构建产物名固定为 `tampermonkey-plugin-koishi-market-ai-helper.user.js`。
2. GitHub Actions 构建后上传 `.user.js` 和 checksum 到 Release assets。
3. Release body 写清安装链接、校验信息、主要变更。
4. 继续用 commit message 控制发布构建，`build release` 只发布 Release。

### 取舍

优点是和现有 Actions 最搭，不需要额外平台账号，每个版本都有归档。

缺点是 latest URL 会重定向，用户发现成本较高，需要 README 或 Release 页面引导。

## 渠道二：GitHub Pages 固定 URL

### 定位

第二优先级，提供固定、直观、可读的 `.user.js` URL，适合作为 `@downloadURL` / `@updateURL` 的主候选。

参考：

```text
https://docs.github.com/en/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages
```

建议 URL：

```text
https://OWNER.github.io/tampermonkey-plugin-koishi-market-ai-helper/tampermonkey-plugin-koishi-market-ai-helper.user.js
```

### 要做

1. GitHub Actions 构建 `.user.js` 后部署到 Pages。
2. README 增加 Pages 安装链接。
3. `vite.config.ts` 预留 `@downloadURL` / `@updateURL`，等 owner 和 Pages 地址稳定后填入。
4. 保持 Release 与 Pages 产物来自同一次构建，避免版本漂移。

### 取舍

优点是 URL 固定、适合自动更新、不依赖 latest release redirect。

缺点是需要启用 GitHub Pages，且 Pages 部署失败时仍要保留 Release 作为备用入口。

### Action 触发

`build publish` 负责完整发布：先创建或更新 GitHub Release，再部署 GitHub Pages。

## 渠道三：Greasy Fork

### 定位

稳定后的主要公开分发渠道，用于提升搜索发现和安装体验。

参考：

```text
https://greasyfork.org/en/help/meta-keys
```

### 当前已补齐

1. `LICENSE`，采用 MIT License。
2. README 中的隐私说明、LLM 请求说明、API key 存储说明、跨域请求域名说明。
3. userscript metadata 的 `@description`、`@namespace`、`@license`、`@homepageURL`、`@supportURL`。

### 发布前仍需确认

1. 构建产物没有本地调试 URL、测试 token 或私人 endpoint。
2. Greasy Fork 页面说明清楚写明第三方 LLM 请求和 API key 存储策略。

### 注意

1. 从 Greasy Fork 安装的脚本通常由 Greasy Fork 自己负责更新。
2. Greasy Fork 会处理部分 metadata，尤其要注意 `@updateURL` / `@downloadURL` 行为。
3. 发布说明必须清楚说明用户输入可能发送到用户配置的 LLM API。
4. GitHub Actions 不建议模拟网页登录上传，后续优先使用平台侧 GitHub sync 或 webhook。

### 取舍

优点是用户发现能力强、安装体验成熟、反馈集中。

缺点是合规和说明成本更高，发布前必须写清第三方请求、隐私和 license。

## 渠道四：OpenUserJS

### 定位

Greasy Fork 之后的备选镜像渠道，也面向 userscript 用户。

Tampermonkey 官方脚本页也把 OpenUserJS 列为脚本来源之一：

```text
https://www.tampermonkey.net/scripts.php
```

### 要做

1. 先在 README 中列为后续计划。
2. Greasy Fork 发布稳定后，再同步 OpenUserJS。
3. 复用 Greasy Fork 准备好的 license、隐私说明和 LLM 请求说明。
4. 检查 OpenUserJS 对 metadata、license、外部请求的要求。
5. 后续优先使用平台侧 GitHub sync，避免 CI 里保存网页会话类凭据。

### 取舍

优点是增加公开镜像，避免绑定单一脚本站点。

缺点是多平台发布会增加维护成本，需要避免平台之间版本漂移。

## 渠道五：GitHub / Gist Raw

### 定位

备用分发方式，不作为主推。

用户可以通过 GitHub Raw 或 Gist Raw 安装 userscript，但版本管理、更新体验和说明入口都弱于 Release / Pages。

Tampermonkey 官方脚本页也提到 GitHub / Gist 可作为脚本来源：

```text
https://www.tampermonkey.net/scripts.php
```

### 适合场景

1. 临时测试构建产物。
2. 分享实验版本。
3. 主渠道异常时提供备用下载。

### 不主推的原因

1. Raw URL 不适合正式版本归档。
2. README、changelog、checksum 的关联较弱。
3. 用户更难判断自己安装的是哪个版本。
4. 当前项目不提交 `dist/`，所以直接 push 源码仓库不会自动产生可安装的 Raw 构建产物。

## GitHub Actions 触发语义

1. `build action`：只运行检查和构建，上传临时 artifact，不发布 Release 或 Pages。
2. `build release`：构建并创建/更新 GitHub Release，只发布 Release assets。
3. `build publish`：构建并完整发布，包含 GitHub Release assets 与 GitHub Pages 固定安装地址。

## 入口一：README 与 Release body 一键安装

### 定位

这不是新的托管渠道，而是用户入口层。目标是让用户不需要理解 Release assets，也能直接安装脚本。

README 建议入口：

```markdown
[一键安装 userscript](https://github.com/OWNER/REPO/releases/latest/download/tampermonkey-plugin-koishi-market-ai-helper.user.js)
```

启用 Pages 后增加：

```markdown
[固定地址安装 userscript](https://OWNER.github.io/tampermonkey-plugin-koishi-market-ai-helper/tampermonkey-plugin-koishi-market-ai-helper.user.js)
```

### Release body 建议内容

1. Install userscript 链接。
2. checksum 链接。
3. 本次主要变更。
4. Tampermonkey / Violentmonkey 安装提示。

## 入口二：Userscript.Zone 搜索发现

### 定位

搜索发现入口，不是发布托管目标。

Userscript.Zone 可以帮助用户按 URL 或域名搜索 userscript。Tampermonkey 官方脚本页列出了 Userscript.Zone：

```text
https://www.tampermonkey.net/scripts.php
```

### 要做

1. 等 Greasy Fork 或 OpenUserJS 页面稳定后，观察 Userscript.Zone 是否能索引到。
2. README 中可补一句“也可通过 Userscript.Zone 搜索 koishi.chat 相关脚本”。
3. 不为 Userscript.Zone 单独做构建或发布流程。

## 当前推荐执行顺序

1. 保留并完善 GitHub Release latest download。
2. 完善 README 与 Release body 一键安装入口。
3. 新增 GitHub Pages 固定 URL 分发。
4. 稳定后发布 Greasy Fork。
5. Greasy Fork 稳定后考虑 OpenUserJS 镜像。
6. GitHub / Gist Raw 只作为备用说明。
7. Userscript.Zone 只作为发现入口记录。

## 当前验收清单

1. `npm run check` 通过。
2. GitHub Actions 能生成 `.user.js`。
3. Release assets 包含 `.user.js` 和 checksum。
4. README 顶部有一键安装入口。
5. Release body 有一键安装入口。
6. Pages 启用后，固定 URL 可以直接安装。
7. 发布 Greasy Fork 前，license、隐私说明、LLM 请求说明齐全。
